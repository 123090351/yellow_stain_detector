### 0.1 当前任务定义已收敛

客户/需求方最新口径：**先不管前序团队旧模型效果，先自己搞一个模型出来看效果，越高越好**。项目现在更像标准机器学习任务：

```text
数据整理 → 人工标注 → train/val/test 划分 → 训练 → 验证/调参 → 测试 → 看错例 → 补数据重训
```

当前最实用的任务定义：

- **主任务**：整图 OK / NG 分类。`OK` = 合格品；`NG` = No Good，不合格品 / 黄斑残次品。
- **实现方式**：优先用 **YOLO 检测模型**检测 `huangban` 框，再将检测结果转成分类：
  - 检出 `huangban` 框且置信度超过阈值 → NG
  - 没有检出 `huangban` 框 → OK
- **辅助任务**：给出黄斑大致位置框用于解释和人工复核。
- **标注形态**：当前用**水平矩形框**即可，不需要一开始做旋转框、多边形或分割。黄条是斜线时，用水平矩形把整条黄条包住，允许包含少量正常背景。

历史上曾倾向 YOLO 分割；最新实操口径已调整为：**分类准确率最重要，位置框能辅助说明即可，所以先走 YOLO detection baseline**。分割、旋转框、异常检测保留为后续效果不够时的备选。

### 0.6 YOLO 基础知识约定

YOLO detection 输出：

```text
类别（huangban）+ 框位置 + 置信度
```

YOLO 标准数据格式：

```text
dataset/
  images/train/
  images/val/
  images/test/
  labels/train/
  labels/val/
  labels/test/
  data.yaml
```

每张图片对应同名 `.txt`：

```text
0 x_center y_center width height
```

坐标均为 0 到 1 的归一化比例，不是像素。多个框多行。OK 图为空 `.txt`。

常用划分：

- 常规：70 / 15 / 15
- 数据多时：80 / 10 / 10 或 90 / 5 / 5
- 当前 47 黄斑 + 22 OK 可先按：
  - train: 33 黄斑 + 16 OK
  - val: 7 黄斑 + 3 OK
  - test: 7 黄斑 + 3 OK

指标解释：

- `precision = TP / (TP + FP)`：模型报黄斑时，有多少是真的。低 precision = 误检多。
- `recall = TP / (TP + FN)`：真实黄斑里，有多少被找到。低 recall = 漏检多。
- `IoU`：预测框与人工框的交并比；完全重合为 1，不重合为 0。
- `mAP50`：IoU >= 0.5 算框对时的检测综合指标。
- `mAP50-95`：IoU 从 0.5 到 0.95 多个阈值下平均，更严格。
- `conf`：推理置信度阈值。降低 `conf` 会减少漏检但增加误检；提高 `conf` 会减少误检但增加漏检。

查看结果时重点看：

- `val_batch0_labels.jpg`：验证集真实标注
- `val_batch0_pred.jpg`：模型预测结果
- `confusion_matrix.png`：真实类别 vs 预测类别
- `results.png` / `results.csv`：训练曲线与指标

## 1. 项目概述

**目标**：开发一个 AI 视觉模型，识别白色海绵胸杯上的「黄斑」（一种偏黄的污渍），将产品判定为**合格品**或**残次品**。

**关键特征**：黄斑比黑点**更不明显**，肉眼需要很仔细才能看出来（信号微弱、边界模糊）。形态上更准确的描述是**黄条/黄线**（条状、线状），而非片状色块。

**最终交付物**：**一个训练好的模型文件，格式为 `.evo`**。客户拿到 `.evo` 后，由他们自己的技术人员负责后续部署、跑批、界面等。
- ⚠️ `.evo` **不是计算机视觉/深度学习的标准格式**（标准格式是 `.pt`/`.onnx`/`.engine`）。它很可能是前序团队那套平台的私有格式，**必须向客户确认它如何生成**（见第 7 节）。
- ⚠️ 「交一个 `.evo`」**不等于**「随便导出一个文件就算完成」。隐含要求是「训练出一个**效果达标**的模型并导出」。成败几乎完全押在模型效果这一个指标上。

**不需要做的事**：不需要 Web 端，不需要做平台/应用。客户最初提过「纯本地一键跑通」，但后来明确交付物只是 `.evo` 模型文件，所以平台基本不在范围内。

**时间**：2 个月，覆盖从标注到模型的全流程。

**成像条件**：受控且稳定（同一相机、同一光照、同一背景，照片质量没问题）。这是重大利好——工业视觉一半的坑来自成像不稳定，本项目绕开了。

**人力**：可能有约 6 名 part-time 同学几周后加入，非理工科、不会 coding，**适合承担标注工作**。但项目计划应按**单人可完成**来设计，把这 6 人当作纯加速项（来了提速标注，不来计划也不塌）。

## 3. 核心难点（按真实难度排序）

1. **黄条的可分性（唯一真正的硬骨头）**：已通过 LAB 实验 + 肉眼观察确认，缺陷分两种形态：
   - **A型（线条状）**：细长斜线，LAB_b 增强图里显示为亮白色条，清晰可见，前序系统能识别。开发者对着 LAB_b 图可以自行标注。
   - **B型（极弱信号）**：颜色极浅，**LAB_b、CLAHE、Deviation 三种处理方法全部试过，均几乎看不出来**，前序系统漏检。注意：B型图上的小方框是**黑点检测器**跑出的结果（置信度极低），与黄条无关。B型是真实缺陷，但**必须由质检专家指出位置才能标注**，开发者无法独立完成。
2. **标注标准的统一（主观性难点）**：「什么算黄条」本身模糊，多人标注必然不一致，直接影响模型上限。靠标注规范 + LAB 增强显影来缓解。
3. **效果不达标时的诊断难点**：训练完效果差时，要能判断到底是——数据标错？信号太弱？分辨率不够丢了细节？超参没调好？正负样本不平衡？这几个原因症状相似但解法全然不同。
4. **工程与代码（几乎不算难点）**：代码量很小（约 100–200 行，大部分是读文件夹、格式转换、写 `data.yaml`、结果循环等胶水代码；核心训练就几行 Ultralytics）。

**结论**：难度集中在少数「判断点」上，工程很轻。2 个月交付「一个能用、效果不错的本地模型 + 导出 `.evo`」是现实的；追平黑点 99% 是 stretch goal，取决于黄斑可分性。

## 4. 推荐技术路线

主要有四条路，建议组合使用：

1. **传统颜色阈值（OpenCV）**：用 LAB/HSV 通道 + 阈值。快、可解释，是极好的 baseline，**有可能直接吃掉很大一部分**。缺点是对光照敏感（本项目成像受控，缓解了这点）。
2. **目标检测（YOLO 检测版）**：画矩形框框出缺陷。前序团队黑点很可能就是这类。但**黄斑是不规则片状，用矩形框会框进大量正常区域，把「黄斑」这个类别稀释得不纯**，效果可能打折。
3. **语义分割（YOLO-seg / 配合 SAM 辅助标注）**：逐像素勾出缺陷轮廓。**对黄斑这种片状污渍更贴合**，是主模型的首选方向。
4. **异常检测（Anomalib 库）**：只用「合格品」训练，模型学会「正常长什么样」，偏离即报异常。**几乎不需要缺陷样本**，适合缺陷样本少/信号弱的场景；缺点是输出是热力图、阈值要调。是强力备选。

**工具链**：用 **Ultralytics YOLO**（底层是 PyTorch）。原因：与前序团队一致（标注工具/数据格式/流程可复用）、文档好、上手快。核心代码骨架：

```python
from ultralytics import YOLO
model = YOLO("yolo11n-seg.pt")                 # 分割版预训练模型（检测版去掉 -seg）
model.train(data="data.yaml", epochs=100, imgsz=1280, batch=8)
results = model("某张图.jpg")                   # 推理
model.export(format="onnx")                     # 导出（.evo 的生成方式待客户确认）
```

> 关键参数：`imgsz`（喂给模型的图片分辨率）——黄斑微弱，**需要喂高分辨率图保住细节**，这是本项目少数需要特别注意的点，也很吃 VRAM。

**当前倾向（根据 LAB 实验结果 + 缺陷形态更新）**：
- ❌ 传统颜色阈值**降级为辅助**（OK 图误报问题确认此路不通作为主方案）
- ✅ 主模型走 **YOLO 分割**（不是检测）：缺陷是条状/线状，矩形框会框进大量正常白色区域稀释信号，分割能贴着条状轮廓标注更准确
- ✅ **Anomalib 异常检测**保留为强力备选（OK 图数量少但够用，且天然规避误判问题）

## 5. ⭐ 关键技术洞察：LAB 色彩空间（整个项目的命门）

黄 vs 白本质是**颜色问题**。把图从 RGB 转到 **LAB 色彩空间**，其中的 **b 通道**专门表示「蓝–黄」轴（越黄值越高）。**一块肉眼几乎看不清的黄斑，在 b 通道里可能会非常明显地「跳」出来。** （HSV 的饱和度通道有类似效果。）

**✅ 实验已完成（2026-06-26）**，结论：
- A型缺陷在 LAB_b 里**非常清晰**（亮白色斜线），OK 图均匀无异常，对比明显
- OK 图整体 b 值（138）略高于黄条图整体均值（136）→ 整图统计无法区分两组，信号被背景淹没 → 纯颜色阈值彻底出局
- B型缺陷：尝试了 LAB_b、CLAHE、Deviation 三种方法，均无法清晰显示 → 信号在噪声底部，是图像处理的极限，不是方法问题

**生成的增强图**：
- `enhanced/LAB_b/` — 原始 LAB b 通道增强图，用于辅助标注
- `enhanced_v2/CLAHE/`、`enhanced_v2/Deviation/` — 针对B型的增强版（47张缺陷）

**脚本**：`enhance_for_labeling.py`、`enhance_v2.py`

## 7. 算力

- **不需要换电脑**。本地机器（含 MacBook Air）只作操作终端，真正吃 GPU 的训练跑在**云 GPU** 上。
- **显存（VRAM）≥ 16GB**（黄斑微弱→需喂高分辨率图→吃显存）。RTX 4090（24GB）性价比好。
- 平台：海外 Vast.ai / RunPod / Lambda，或国内阿里云 / 腾讯云 / 火山引擎。选**按量付费、随开随关**的实例。
- 成本量级：整个项目累计 GPU 使用可能就几十小时，约**几百元人民币**。这笔钱应由客户出（客户已表示可提供显卡）——让客户提供带 NVIDIA 卡的云服务器或报销云 GPU 费用，比寄物理硬件更省事。
- 误区澄清：「等训练」不占用有效时间——训练在云端后台跑，期间可以同时标注/看数据/分析；单次训练（几百张图、YOLO）大概率只需几十分钟到一两小时。
